# 프로토타입 워크스페이스 — 작업 규칙

이 저장소는 **여러 서비스/화면의 프로토타입 + 개발 명세(기획서)** 를 한 워크스페이스에서 만든다. 각 프로토타입은 **자체 HTML 파일**로 격리되며, 모두 동일한 구조(좌·중앙 프로토타입 + 우측 440px 명세 사이드 패널)와 동일한 명세 템플릿(DEV-SPEC v1)을 따른다. Tailwind CDN + 인라인 CSS/JS만 사용한다(빌드 없음).

---

## 1. 파일 구조

- 각 프로토타입은 **자체 HTML 파일 한 개**로 완결한다. 새 화면을 추가할 때는 새 파일을 만들고, 기존 프로토타입을 덮어쓰지 않는다.
- 현재 보유한 프로토타입
  - `index.html` — **전남아이톡** 메인 화면 (전라남도 출산·육아 통합 서비스, MAIN-001)
- 새 프로토타입을 만들 때 파일명은 `<서비스영문명>.html` 또는 `<서비스명>-<화면명>.html` 형태로 짓는다 (예: `italk-mypage.html`).
- 한 파일 내 영역 구분
  - 좌·중앙 = **프로토타입** (사이드바·헤더·메인 또는 풀폭 랜딩 등 자유)
  - 우측 440px 패널 = **개발 명세(기획서)**, `id="spec-panel"`
- 명세 본문 기본값은 `DEFAULT_SPEC_HTML` 상수(템플릿 리터럴)에 들어 있다. 명세를 수정하라는 지시는 이 상수를 수정한다는 뜻이다.

---

## 2. 기획서 = "개발 명세(DEV-SPEC v1)" 표준 템플릿

기획서는 **개발자(FE/BE)가 코드를 짜는 데 필요한 정보만** 담는다. 기획자 중심 서술(목적·페르소나·디바이스 정책·디자인 토큰·ASCII 레이아웃)은 넣지 않는다. 모든 화면 명세는 동일한 6개 섹션 구조를 따른다.

### 표준 섹션 (고정 순서)

| # | 섹션 | 내용 |
|---|---|---|
| 1 | 화면 개요 | **고정 7항목 표** (아래 표준 개요 항목 참조) |
| 2 | 사용자 플로우 | 2.1 Happy Path (번호 ol) + 2.2 에러/엣지 (케이스/처리 2열 표) |
| 3 | 유효성 검증 | 필드 / 제약 / 위반 시 처리 (3열 표). 조회 화면이면 응답 데이터 검증으로 대체 |
| 4 | 상태 / Enum 정의 | 4.1 카테고리, 4.2~ 상태 enum (값/라벨/배지), 마지막 4.X "공통 배지 색상" |
| 5 | 권한 매트릭스 | 액션 × 역할 표 (O / X / 읽기만 / —) |
| 6 | 변경 이력 | 버전/일자/작성자/변경 내용 표 |

### 표준 개요 항목 (고정 7행 — 라벨·순서 절대 변경 금지, **추가 금지**)

화면이 어떤 유형이든 ① 화면 개요 섹션은 아래 7개 행을 **정확히 이 순서로, 이 라벨로** 사용한다. 값만 화면별로 다르고 라벨·순서·항목 수는 모든 화면이 동일해야 한다.

**절대 규칙**
- 이 7개 행 외의 어떤 항목도 추가하지 않는다 (라우트 / 레이아웃 유형 / 의존 모듈 / SEO / 관련 화면 / 디자인 시안 링크 등 일체 금지).
- 행을 빼지도 않는다. 해당 화면에 값이 없으면 `—` 또는 `해당 없음`으로 채운다.
- "이번 화면은 특별하니 한 줄만 추가" 식의 예외도 만들지 않는다. 부가 정보가 필요하면 ② 사용자 플로우 또는 ③~⑥ 다른 섹션에서 다룬다.

| 순서 | 항목 | 값 가이드 |
|---|---|---|
| 1 | 화면 ID | 코드 (예: `MAIN-001`, `MYPAGE-002`) — `<서비스약어>-<3자리>` 패턴 |
| 2 | 화면명 | 한글 명칭 (예: 메인 화면, 마이페이지, 결제 상세) |
| 3 | 상위 메뉴 | IA 경로 또는 "랜딩 (상위 없음)" / "GNB > 지원사업 > 신청내역" 형식 |
| 4 | 진입 경로 | 어떻게 도달하는가 (직접 URL · 외부 링크 · 로그인 후 자동 리다이렉트 · 모달 트리거 등 콤마 구분) |
| 5 | 접근 권한 | 역할/인증 상태 — `spec-tag` 사용 (`anonymous` / `member` / `verified` / `admin` 등) |
| 6 | 우선순위 | `P0` / `P1` / `P2` — `spec-tag.required`(P0) / `spec-tag.optional`(P1+) |
| 7 | 디바이스 | `PC` / `모바일` / `반응형 (PC / 모바일, breakpoint 768px)` |

### 절대 넣지 않는 섹션

- 컴포넌트 명세 (Props/이벤트/데이터 모델 별도 카드)
- API 명세 (엔드포인트/요청·응답 스키마/에러 코드 표)
- 인수 조건 (Given-When-Then)
- 비기능/분석 (성능 목표, GA 이벤트, 접근성, i18n)

이 4개는 사용자가 명시적으로 의뢰 시점에 다른 곳에 작성하기로 했으므로 기획서에 자동으로 추가하지 않는다.

### 형식 규칙

- 본문 최상단에 표준 안내 박스(파란 좌측 보더, `#f8fafc` bg)를 둔다.
- 메타 정보(`<div class="spec-meta">`)에는 화면 ID, 라우트, 버전, 최종 수정일, 대상, 템플릿 버전(`템플릿 DEV-SPEC v1`)을 표시한다.
- 섹션 제목은 `<h2><span class="num">N</span>제목</h2>` 형식.
- 하위 절은 `<h3>N.X 제목</h3>` 형식.
- 강조 태그: `spec-tag.required`(빨강 — P0/필수), `spec-tag.optional`(파랑 — 옵션), `spec-tag.todo`(노랑 — TODO/추후), `spec-tag.ceo`(보라), `spec-tag.manager`(초록).
- 코드 식별자/짧은 enum 값은 `<code>`, 다줄 코드/타입 정의는 `<pre><code>`.
- 새 화면 명세를 만들 때 해당 화면에 없는 섹션은 **삭제하지 말고 "해당 없음"으로 명시**한다(누락이 아님을 보이기 위함).

## 3. 프로토타입 + 기획서 내보내기

우상단 "내보내기" 버튼은 **프로토타입 전체 + 기획서 사이드 패널**을 단일 HTML 파일로 다운로드한다 (`safenow-dashboard-YYYY-MM-DD.html`).

### 동작 규칙 (수정 시 유지)

- `document.documentElement`를 클론해서 프로토타입 전체 구조(사이드바·헤더·메인·기획서 패널) 보존.
- 기획서 본문은 현재 편집 결과(`specBody.innerHTML`)를 인라인으로 박고 `contenteditable` 해제 → 받는 사람에게는 **읽기 전용**.
- 받는 사람에게 무의미한 컨트롤은 제거: `#spec-mode-btn`, `#spec-export-btn`, `#spec-reset-btn`, `#spec-status`. 헤더의 "기획서" 토글(`#spec-toggle`)만 남긴다.
- 기본은 **패널 닫힘** (`is-hidden` 추가, 토글 버튼 `is-active` 제거). 개발자가 전체 화면으로 플로우를 보다가 토글로 열도록.
- 본문(`<body>`) 안의 인라인 스크립트는 모두 제거하고, **정적 뷰 전용 최소 스크립트**(추이 차트 렌더 + 6/12 토글 + 기획서 패널 토글)만 새로 주입.
- `<head>`의 Tailwind config 스크립트는 **반드시 보존** — 제거하면 `text-primary-*`, `bg-sidebar` 같은 커스텀 색이 깨진다.
- 외부 의존: Tailwind CDN, Pretendard 폰트 CDN. 인터넷 필요.

---

## 4. 코드 / 스토리지 컨벤션

- localStorage 키는 **프로토타입별로 네임스페이스를 분리**한다.
  - 패턴: `<service>_spec_doc_v{n}`, `<service>_spec_panel_open`
  - 현재 사용 중
    - 전남아이톡: `italk_spec_doc_v3`, `italk_spec_panel_open`
  - **템플릿을 변경할 때마다 해당 서비스의 `_doc` 버전을 올린다** (구버전 저장본 무효화 → 사용자가 새 템플릿을 자동으로 보게).
- 스타일 클래스는 `spec-` 접두로 격리되어 있다(`spec-panel`, `spec-card`, `spec-tag`, `spec-meta`, `spec-ascii` 등). 새 스타일을 추가할 때도 같은 접두를 쓴다.
- 인라인 스크립트는 끝의 큰 IIFE 한 곳에 모은다. 새 모듈을 만들 필요 없음.

---

## 5. 작업 시 주의

- 기획서 템플릿을 바꿀 때는 **반드시 `SPEC_STORAGE_KEY` 버전을 올린다**. 안 그러면 사용자가 새로고침해도 구버전 저장본만 보인다.
- 기획서 본문은 한국어로 작성한다.
- 마케팅성 문구·이모지·과한 형용사 금지. 개발자가 코드/스키마를 도출할 수 있는 사실만 적는다.
- 프로토타입 UI에 카드를 추가/제거하면 기획서의 ②사용자 플로우, ③유효성 검증, ⑤권한 매트릭스도 함께 업데이트한다.
